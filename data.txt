## 1Ô∏è‚É£ Recommended Flow

### **Step 1: Registration**

* All users register as `user` (normal user).
* Default permissions: read-only or limited.

### **Step 2: Role Switch Request**

* In **Settings**, allow users to pick a new role: `doctor` or `pharmacy`.
* Once they choose, show **role-specific form** dynamically.

  * Doctor form: qualifications, specialization, license number, clinic info, etc.
  * Pharmacy form: pharmacy name, license, address, contact, etc.
* Do **not switch their role in the DB yet**. Instead, create a **pending request**.

---

### **Step 3: Role Request Submission**

* User submits form ‚Üí backend creates **role request document**:

```js
{
  user: ObjectId("..."),
  requestedRole: "doctor",
  data: { ...full info... },
  status: "pending",
  submittedAt: Date
}
```

* This keeps the main user entry intact as `role: "user"`.

---

### **Step 4: Admin Verification**

* Admin has a dashboard to **approve/reject requests**.

* On approve:

  * Update user‚Äôs `role` to `doctor` or `pharmacy`.
  * Create/attach doctor/pharmacy profile with all info.
  * Set `isVerified: true`.

* On reject:

  * Notify user with reason.
  * Role remains `user`.

---

### **Step 5: Post-Verification**

* Verified users now have full permissions.
* Use **middleware** like `checkDoctorVerification` and `checkPharmacyVerification` to protect routes.
